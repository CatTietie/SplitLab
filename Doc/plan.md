## 多层互斥流量模型

```
桶空间: 0 ─────────────────────────────────────────── 9999

Layer A (salt="abc"):
  ┌─── Exp 1 ───┐┌─── Exp 2 ───┐┌── 未分配 ──┐
  │  0 - 4999   ││ 5000 - 7499 ││ 7500 - 9999│
  └─────────────┘└─────────────┘└────────────┘

Layer B (salt="xyz"):  ← 不同 salt，同一用户得到不同桶号
  ┌────────── Exp 3 ──────────┐┌─── Exp 4 ───┐
  │        0 - 6999           ││ 7000 - 9999 │
  └───────────────────────────┘└─────────────┘

用户 hash:
  hash("user_1" + "abc") % 10000 = 3721 → Layer A: Exp 1
  hash("user_1" + "xyz") % 10000 = 8502 → Layer B: Exp 4
  ✓ 层间独立, 层内互斥
```

## 性能评估 (10万用户)

### 负载估算

| 组件 | 计算 | QPS/吞吐 | 单节点容量 | 余量 |
|------|------|----------|-----------|------|
| 配置接口 | 100K ÷ 30s | ~3,333 QPS | Redis: 100K+ QPS | 30x |
| 分流决策 | 100K × N次/天 | SDK本地 | 微秒/次 | ∞ |
| 事件入库 | 10 ev/user/day = 1M/day | ~12 QPS | PG: 5K+ INSERT/s | 400x |
| 统计查询 | PM按需查看 | <10 QPS | 聚合<100ms | 充裕 |

### 瓶颈分析

**结论: 10万用户规模下无瓶颈。** 所有组件远未达到单机极限。

潜在的性能瓶颈只在 **10M+ 用户** 时出现:
1. **事件表膨胀** (>1亿行): 需按 experiment_id 分区 + 日期分区
2. **统计查询变慢**: 引入物化视图或预聚合表
3. **配置接口高并发**: 多后端实例 + Redis pub/sub 同步失效

### 优化路线图

```
10万用户: 单机 PG + Redis (当前架构) ✓
100万用户: PG 读写分离 + 事件表按月分区
1000万+: 事件存储迁移 ClickHouse + 后端水平扩展
```

## 交付验证 SLA

| SLA 指标 | 要求 | 验证方式 | 测试文件 |
|----------|------|----------|----------|
| 分流偏差 | < 0.5% | 卡方检验 + 最大偏差断言 (100K用户) | `sdk/tests/test_traffic_split.py` |
| 配置热更新 | < 60s | 修改配置 → SDK 感知时间 | `sdk/tests/test_hot_reload.py` |
| 事件零丢失 | 0 丢失 | 模拟崩溃 → WAL恢复 → 全部送达 | `sdk/tests/test_zero_loss.py` |
| 生命周期 | 全流程 | create→start→track→stats→pause→rollout | `sdk/tests/test_e2e.py` |
